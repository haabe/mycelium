# JiT Tooling: Metrics Source Detection

Mycelium detects which external metric sources apply to this product and configures them, the same way `detector.md` detects the tech stack. The goal: replace manual "I checked GitHub" / "I looked at Stripe" reports with timestamped, sourced, diffable snapshots at L0 (purpose), L1 (strategy), L2 (opportunity), and L5 (market).

This file is the detection logic. The unified pull entrypoint is `/metrics-pull`. Per-source logic lives in `metrics-adapters/<source>.md` (and is generated on demand — see `metrics-adapters/GENERATING.md`).

## When This Runs

- During `/interview` Phase 6 (alongside tech-stack detection).
- Explicitly via `/metrics-detect` (for retrofit — projects that started before v0.14).
- Before `/metrics-pull` when `active-metrics.yml` is missing.

## Detection Process

### Step 1: Scan for Metric Source Signals

| Signal in project | Implied source | Class | Credential requirement |
|-------------------|----------------|-------|------------------------|
| `git remote` points to `github.com` | `github` | traffic | `gh` CLI authed with write scope on the repo |
| `git remote` points to `gitlab.com` / self-hosted GitLab | `gitlab` | traffic | `glab` CLI or `GITLAB_TOKEN` env |
| `fastlane/`, `ios/` dir with App Store ID | `app_store_connect` | reviews | API key file path via env (`APP_STORE_CONNECT_KEY_PATH`) |
| `android/` with Play package ID | `play_console` | reviews | Service account JSON via env (`GOOGLE_APPLICATION_CREDENTIALS`) |
| `plausible-tracker` in deps, or `PLAUSIBLE_DOMAIN` env | `plausible` | traffic | `PLAUSIBLE_API_KEY` env |
| `posthog-*` SDK in deps | `posthog` | traffic + events (two adapters) | `POSTHOG_API_KEY` env |
| `fathom-client` in deps | `fathom` | traffic | `FATHOM_API_KEY` env |
| `@vercel/analytics`, `vercel.json` with analytics | `vercel_analytics` | traffic | `VERCEL_TOKEN` env |
| `gtag`, `ga-*`, `G-` measurement id in HTML/env | `google_analytics` | traffic | GA4 service account |
| `stripe` SDK in deps + `STRIPE_SECRET_KEY` env | `stripe` | events | `STRIPE_SECRET_KEY` env (read-only recommended) |
| `@sentry/*` SDK in deps | `sentry` | events | `SENTRY_AUTH_TOKEN` env |
| `intercom-*` SDK in deps | `intercom` | support | `INTERCOM_ACCESS_TOKEN` env |
| `zendesk` SDK in deps | `zendesk` | support | `ZENDESK_API_TOKEN` + subdomain env |
| Linear workspace URL in CLAUDE.md or config | `linear` | support | `LINEAR_API_KEY` env |
| GitHub repo (any) | `github_issues` | support | same as `github` |
| CLAUDE.md / README mentions a product URL but no analytics SDK detected | `web_traffic` (generic) | traffic | ask user which analytics provider |

### Step 1b: Ask, Don't Assume

Metric sources depend on where the product LIVES, not only what's in the repo. After the signal scan, ALWAYS ask:

1. "Is there a deployed product URL I should track analytics for?"
2. "Do you sell anything — Stripe, Paddle, Lemon Squeezy, Gumroad?"
3. "Is the product on any app store?"
4. "Where do users send feedback — Intercom, Zendesk, email, a form?"

Record user-declared sources even if no signal in the repo matched. The detector output is the union of detected and declared sources.

### Step 2: Confirm Each Source

For each candidate source, confirm individually with the user:

> "I see `plausible-tracker` in your dependencies, so I'd like to track Plausible metrics. The adapter needs `PLAUSIBLE_API_KEY` in your environment. Do you have that set up? Track Plausible?"

Accept: yes / no / later. Sources marked "later" are saved as `status: deferred` and re-asked on the next `/metrics-detect`.

### Step 3: Ensure Adapters Exist

For each confirmed source, check `metrics-adapters/<source>.md`:

- **Exists**: good. Note the `last_known_working` date; flag if >180 days.
- **Missing**: invoke the adapter-generation workflow in `metrics-adapters/GENERATING.md`. The agent generates the adapter using the template + GitHub reference + vendor docs (via `mcp__context7` / `WebFetch`), then presents it for user confirmation before saving.

Mycelium ships only `github.md` as the reference adapter. All other adapters are generated on demand during this step.

### Step 4: Write active-metrics.yml

Output to `.claude/jit-tooling/active-metrics.yml`:

```yaml
# Auto-generated by /metrics-detect. Gitignored by default — the target
# identifiers are not secrets but snapshots may reveal business state.
# If you DO want this committed (e.g., for a dogfood repo), un-gitignore
# both active-metrics.yml and evals/metrics/ explicitly.
detected_at: "YYYY-MM-DDTHH:MM:SSZ"
confirmed_by_user: false    # Set to true after the user confirms the full set

sources:
  github:
    status: "active"         # active | deferred | disabled
    target: "owner/repo"
    window_days: 14
    adapter_version: 1
    last_pulled_at: null
  plausible:
    status: "active"
    target: "mycelium.dev"
    window_days: 30
    adapter_version: 1
    last_pulled_at: null
  stripe:
    status: "deferred"       # user said "later"
    target: "prod"            # environment label, not a secret
    adapter_version: 1
```

`active-metrics.yml` is project-scoped (all diamonds in the project share it). See `active-metrics.example.yml` for a full example.

### Step 5: Confirm with User

After writing the draft, show the user the full source list and ask:

> "I've configured N metric sources: [list]. Ready to pull? Run `/metrics-pull` to fetch the first snapshot. I'll remind you before `/diamond-assess` at L0/L1/L2/L5 if the latest snapshot is older than 7 days."

Set `confirmed_by_user: true` only after this confirmation.

## Source Classes

Adapters declare one of four classes; downstream delta/report logic keys off the class.

| Class | Examples | Native schema fields |
|-------|----------|----------------------|
| `traffic` | GitHub, Plausible, Google Analytics, Fathom, PostHog (pageviews) | `primary_counts`, `traffic`, `referrers`, `top_paths`, `custom_signals` |
| `events` | Stripe, Segment, Sentry, Postgres audit logs | `primary_counts`, `custom_signals` (no `referrers`/`top_paths` — they're traffic-native) |
| `reviews` | App Store Connect, Play Console, Trustpilot | `primary_counts`, `custom_signals`; MAY repurpose `referrers`/`top_paths` with documented `field_interpretations` |
| `support` | Intercom, Zendesk, Linear, GitHub Issues | `primary_counts`, `custom_signals`; MAY repurpose `referrers`/`top_paths` with documented `field_interpretations` |

**Rationale**: earlier drafts treated `referrers` and `top_paths` as universal. The Stripe assumption test (2026-04-16) revealed these are traffic-native; forcing event-shaped sources to emit empty arrays was a design smell. The App Store test then validated that `reviews` can reuse them under repurpose-with-documented-interpretation, giving reviews-class adapters a structured path into the shared downstream pipeline. The `support` class follows the same pattern but is NOT yet validated against a real source — remaining risk, flagged in the v0.14 spec.

## What Mycelium Ships vs Generates

**Ships** (committed to the framework, safe to replace on upgrade):
- `metrics-detector.md` (this file)
- `metrics-adapters/TEMPLATE.md`
- `metrics-adapters/GENERATING.md`
- `metrics-adapters/github.md` (reference adapter)
- `.claude/skills/metrics-pull/SKILL.md`
- `.claude/skills/metrics-detect/SKILL.md`

**Generates per-project** (gitignored by default — user opts in to committing):
- `active-metrics.yml`
- Any other `metrics-adapters/<source>.md` produced by `GENERATING.md`
- Snapshots in `.claude/evals/metrics/<source>/YYYY-MM-DD.json`

## Freshness / Staleness Rules

- `active-metrics.yml` itself: re-run `/metrics-detect` if >90 days old or if dependencies change.
- Per-adapter `last_known_working`: if >180 days, regenerate via `GENERATING.md` rather than patching.
- Per-snapshot: `/canvas-health` flags the snapshot as stale if newest snapshot is >7 days old for L0/L1/L2/L5 diamonds.

## What NOT to Do

- Do NOT store credentials in `active-metrics.yml` or in any adapter file. Mycelium is not a secrets manager. Environment variables and vendor CLIs are the only supported paths.
- Do NOT auto-pull during `/metrics-detect`. Detection configures; `/metrics-pull` fetches. Keep the steps separate so the user can review the source list before any network calls.
- Do NOT fall back silently when a target is missing. Abort and ask.
- Do NOT add a source without a corresponding adapter. If generation fails, mark the source `status: disabled` with a reason.
